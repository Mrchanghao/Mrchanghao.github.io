Database — 소개 (Part 1)

데이터베이스, DBMS가 무엇이고 왜 사용하는지?

이 글은 데이타 베이스에 관한 긴 시리즈의 학습서이다, 우리는 앞으로 아래 내용을 다룰 예정입니다.

  • 데이타베이스에 관한 소개(DBMS가 무엇이고 왜 사용하는지)
  • 데이타베이스에 관한 기초
  • 데이타베이 디자인 과정
  • Normalization(정규화라고 해야 하나여? ㅠㅠ)
  • SQL(the structured query language)
  • Indexing, transaction & stored procedures.
  • 데이타 베이스의 서로 다른 종류들

데이타베이스(이후 부턴 DB라고 부르겠습니다)를 공부하기 전에 DB가 제공하는 기능 대신에 문제를 해결하기 위해 DB가 필요한 이유부터 시작하겠습니다. “그래서, 이 문제들은 무엇인가요?”

왜 우리는 DB가 필요한가?

우리에게 어떠한 데이타들이 있을때, 그리고 이 데이타들을 어딘가에 저장하고 싶을때. 이 데이타들은 어떠한 것도 될 수 있습니다. 이 데이타들은 고객들에 관한 것도 될 수 있고, 상품, 종업원, 고객주문 상황 등 무수한 데이타들이 있을 수 있고 이 데이타들은 텍스트 형식이나, 숫자, 날짜, 다큐먼트 파일, 사진, 오디오나 심지어 비디오 형식으로도 존재 할수 있습니다.

아마도 여러분이 여러분의 회사의 고객정보 데이타를 가지고 있다고 했을 때 처음 떠오르는 것은 스프레드 시트를 열고, 저장하려는 데이터를 무엇이는 작성하는 것입니다.

거기에는 고객들의 이름과 아이디, 위치 등이 있을 테고, 여러분은 거기에 고객정보를 추가로 작성할 것입니다. 후에는 데이터 중 어떤 것들은 지우거나 그것들을 심지어 고칠 수도 있고요.

이젠 여러분이 만족하는 방식으로 스프레드 시트에 데이타가 저장되었습니다. 그걸로 되었을 수도 있습니다. 하지만 그냥 데이타만 가지고 있는 것으로는 DB를 필요로 하기 불충분 하다고 할 수 있습니다. 문제는 그 다음입니다. 엄청나게 많은 잠재적 문제점들이 산재해 있습니다.

만약에 엄청나게 많은 데이타들을 가지고 있다 가정한다면, 예를 들어 1만명의 고객정보라고 해봅시다. “여러분은 스프레드 시트에서 9999명의 고객을 찾기 위해 스크롤을 계속 이동 하시겠습니까?!”, 만약에 보안도 걱정이라면, “다른 누군가가 당신의 데이타에 액세스 할 수 있다면 신경이 쓰이지 않으시겠습니까?”, 만약에 실수로 중복정보를 입력한다면요? “스프레드시트에 정보가 복사되도 괜찮나여?”

이 문제점들은 우리에게 다음과 같은 질문을 던집니다. “언제 우리가 실제로 데이타베이스가 필요한가?”
아래의 잠재적 문제점들을 고려해 보세요:

Size

여러분이 수천명 아니 수백만명의 고객이나 정보를 가지고 있을 수 있습니다.

Accuracy(정밀도)

만약 누군가가 정확하지 않은 데이타를 입력했다면? 만약 그렇다면 실제로 아무것도 스프레드 시트에 잘못된 데이터를 입력하지 못하게 할 수 있습니다.

security(보안)

만약 민감한 데이터라면 여러분은 데이터 접근에 제한을 둘 필요가 있습니다; 그리고 그걸 누구와도 공유할 필요가 없습니다. 게다가 “누군가가 데이터에 변화를 줄 때마다 그걸 여러분이 알아야 할 필요가 있나요?”

redundancy(중복성)

만약 데이터가 중복된 거라면 충돌이 날 수도 있고, 여러분은 반복되지 않은 하나의 데이터만 필요할 뿐입니다.

importance(중요성)

“만약 통신연결이 끊어지거나 충돌로 인하여 데이터를 잃어버린다면?” 여러분은 아마 고통을 느껴봤을 것입니다. 그리고 중요한 데이터를 잃어버리는 건 말이 안 됩니다. 예를 들어 고객의 오더나 환자들의 알레르기 기록, 비행기 예약 등을 말이다.

overwriting(덮어쓰기)

한명이 넘는 사람들이 같은 데이터를 같은 시각에 덮어 쓴다면 어떨까? 10명이나 100명이 같은 시각에에 말이다. 아마 모든 사람이 다른 사람들의 변경 사람들을 덮어쓰게 될 것이다.

이런 문제들 증 하나라도 ‘YES! 라고 대답했다면 데이터를 신뢰할 수 있고 안전하게 유지 보수 할 수 있어야 합니다. 따라서 데이터 베이스가 있어야 한고 이것이 우리가 여기서 알아 볼 내용들입니다.

무엇이 데이터베이스(DB)인가?

여러분의 문제점의 중요성에 따라 달라지기 떄문에 데이터에 규칙을 주고 데이터를 저장하는 구조화된 시스템입니다. 만약에 문제점이 사이즈라면 다른 누군가는 민감한 데이터를 문제점으로 가지고 있을 수 있습니다.

이것들은 백그라운드에선 볼수 없는 것 들입니다.; 보안, 완전히 보전된 데이터, 빠르고 안정적으로 접근 할 수 있는 능력, 견고성; 동시에 많은 사람들에게 서비스를 제공하고 심지어 데이터를 손상시키지 않고 충돌 및 하드웨어 문제를 올바르게 해결할 수 있습니다.

그리고 우리가 여기서 필요로 하는 것은; 우리 데이터 구조를 이해하고 이 규칙들을 설명하고, 이 모든 보이지 않는 것들이 실제로 일어 난다는 것입니다.

Database Management System(DBMS)

우리는 자주 DB들을 실수로 Oracle, MySQL, SQL Server, MongoDB라고 부르곤 합니다.
하지만 이것들은 DB가 아니 DBMS라고 불러야 합니다.

DBMS는 우리 개인 컴퓨터나 서버에 설치한 소프트웨어이고 그러고 나서 여러분이 하나 이상의 DB를 핸들링 하는데 사용됩니다.

DB는 실제 데이터와 데이터에 대한 규칙들을 가지고 있고 반면 DBMS는 우리 실제 데이터들을 매니징 하거나 데이터들을 둘러 싸는 프로그램입니다. 그리고 DBMS는 데이터에 여러분이 스펙으로 정한 일들을 하게 합니다. 예를 들어 데이터 타입이 스트링 타입인지 아닌지 아니면 데이터들간의 관계를 판단하는 일들 일 수 있습니다.

실제로는 여러 데이터베이스를 가지는 것이 일반적입니다. 주문과 고객정보를 다루는 DB는 HR 정보를 다루는 DB와는 완전히 독립적일 수 있습니다. 그리고 많은 조직에서 여러 DB를 가지지 않고 여러 DBMS를 가지고 있습니다. 때때로 하나의 DBMS가 다른 것들 보다 나은 점이 있기 때문입니다(??)

여러 DBMS가 있고 아래와 같이 분류 할 수 있습니다.

  • 관계형 DBMS(relational DBMS)
  • Hierarchical DBMS
  • Network DBMS
  • Object-Orient DBMS

  • NoSQL DBMS

    우리는 관계형 DBMS에 집중을 해 보려고 한다 왜??

  • 제일 일반적으로 많이 쓰인다.

  • 우리가 여기에서 이야기할 원리들이 대부분의 플랫폼에 걸쳐서 사용 가능합니다.

  • 만약 바로 NoSQL Database 분야로 뛰어든다면 대부분 당신이 관계형 DB의 개념을 이해하고 있다고 있다고 가정하고 이러한 개념들을 사용하여 NoSQL database가 제공하는 것들을 설명한다. (????)

    RDMS는 Oracle, MySQL, SQL Server, SQLite, DB2, … 등등이 있습니다
